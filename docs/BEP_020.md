## Abstract

Conventions for encoding client identification and version in the 20-byte peer ID field.

---

## Common Encoding Styles

### Azureus Style

Format: `-XXYYYY-ZZZZZZZZZZZZ`

| Position | Length | Description |
|----------|--------|-------------|
| 1-2 | 2 chars | Client identifier |
| 3-6 | 4 digits | Version number |
| 8-19 | 12 chars | Random |

Example: `-AZ2060-` = Azureus 2.0.6.0

### Shadow Style

Format: `XXXXX-----YYYYYYYYYYYY`

| Position | Description |
|----------|-------------|
| 1 | Client identifier (T = BitTornado, S = Shadow, etc.) |
| 2-6 | Version number (up to 5 characters, padded with dashes) |
| 8-19 | Random |

Version characters: `0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz.-`

Example: `S58B-----` = Shadow's 5.8.11

### Mainline Style

Format: `M4-20-8-` followed by random bytes

- `M` prefix
- Version as digits with dashes (major-minor-tiny)
- Remaining bytes random

Examples:
- `M4-3-6--` = Mainline 4.3.6
- `M4-20-8-` = Mainline 4.20.8

### Other Styles

**BitComet**: `exbcXXYY...` where XX.YY is version

**XBT**: `XBT###d-` or `XBT###--`

**Opera**: `OP####...` where #### is build number

**MLdonkey**: `-ML#.#.#-`

---

## Recommendations for New Clients

1. Use the Azureus-style encoding
2. Register a unique two-character client identifier
3. Use four digits for version
4. Leave remaining bytes random
